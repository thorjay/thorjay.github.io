拓展

SSL与SSH

都是用于加密通信的协议

相同点：

  • 加密通信：保护数据在传输过程中的机密性和完整性，通过加密通信来防止第三方未经授权的访问或数据篡改。

  • 密钥交换：都使用密钥交换协议来建立安全连接。SSH通常使用Diffie-Hellman协议，而SSL使用TLS握手过程。

  • 双向认证：都支持双向认证，这意味着客户端和服务器都可以验证对方的身份，确保连接的安全性。

区别：

SSH SSL/TLS
作用 用于安全远程连接另一台机器以发送命令，提供一个加密终端会话 用于在双方之间安全传输数据，通常是网站访问者和网站服务器的在应用层的安全通信，比如HTTPS,WSS,SMTPS,FTPS…
默认端口 22 443（HTTPS）&465（SMTPS）
层次 应用层协议 传输层协议
身份验证 要求客户端用户名&密码或加密密钥 只需服务端认证(客户端不需要)
交互性 基于网路隧道 基于数字证书(SSL证书)，不提供终端或交互性
定义 加密网络协议 安全协议
概念 Secure Shell Secure Sockets Layer

openssl

OpenSSL（Open Secure Socket Layer）是一个开源的密码库，提供了一组用于处理安全通信的加密和解密功能，以及用于生成和管理数字证书的工具。它是一个广泛使用的库，用于加密和安全通信的许多应用程序和协议。

主要功能及作用：

  1. 加密和解密支持： OpenSSL 提供了多种加密和解密算法，包括对称加密（如AES、DES）、非对称加密（如RSA、DSA）、散列函数（如SHA-1、SHA-256）和消息认证码（MAC）等。这些功能使得 OpenSSL 能够保护数据的机密性和完整性。
  2. SSL/TLS 实现： OpenSSL 提供了 SSL（Secure Sockets Layer）和 TLS（Transport Layer Security）协议的实现，用于建立安全的网络通信。这使得 OpenSSL 能够支持安全的 Web 通信（HTTPS）以及其他网络协议的安全版本，如SMTPS、FTPS等。
  3. 数字证书和公钥基础设施（PKI）： OpenSSL 包括工具和库，用于生成和管理数字证书、私钥和公钥。这对于创建自签名证书、为服务器配置 SSL/TLS 和进行身份验证非常有用。
  4. 密码学功能： OpenSSL 提供了丰富的密码学功能，包括随机数生成、伪随机数生成、大数运算和数字签名等。这些功能为安全通信提供了基础支持。
  5. 跨平台： OpenSSL 是一个跨平台的库，可在多种操作系统上运行，包括Linux、Unix、Windows等。这使得它适用于各种不同的应用程序和环境。
  6. 开源： OpenSSL 是一个开源项目，使用 OpenSSL 的源代码可以在 OpenSSL 许可证（类似于双重许可证）下自由使用和分发。
  7. 广泛使用： OpenSSL 在互联网上广泛使用，包括在 Web 服务器、电子邮件服务器、VPN（虚拟私人网络）软件、网络设备等中，用于加密和保护通信。许多知名的互联网服务和应用程序都依赖于 OpenSSL 来确保数据的安全性。

尽管 OpenSSL 在安全通信领域具有广泛的应用，但它也因其复杂性而容易引发安全漏洞。因此，用户和开发人员应定期更新 OpenSSL，并遵循最佳的安全实践来保护其应用程序和系统。

PEM和DER

PEM（Privacy-Enhanced Mail）和DER（Distinguished Encoding Rules）都是用于表示和编码数据的格式标准，它们通常与加密和安全领域有关。

PEM 是一种基于文本的编码格式，通常用于表示各种数据，包括数字证书、私钥、公钥、证书请求和其他加密相关的数据。PEM 格式使用 Base64 编码将二进制数据转换为文本，同时包含头部和尾部标记，以标识数据类型。PEM 文件通常使用扩展名 .pem，.crt 或 .key。PEM 格式是人类可读的，容易查看和编辑，因此在许多场景下广泛使用，如证书文件、密钥文件和配置文件。文件实例：

介绍

最近有几个场景需求都牵涉到rebase这个操作，所以有必要对rebas做一个总结和进一步学习。

场景

1.修改提交记录

2.合并提交记录

3.合并分支

概念

rebase:变基

merge:合并

commit:提交

fast-forward merge:快进合并

branch:分支

Head:这个翻译是头部；我的理解是当前活动状态的游标或指针；可以是当前分支的头部，也可以是某一个提交的指向…

topicbranch：目标分支

basebranch：基础分支

其中，需要理解一个概念快进合并()

什么是变基？

要理解变基，先理解一个更常见的概念”合并(Merge)”。

合并很好理解，就是直接将2个分支合并到一个分支上，其实变基的最终目的也是一样。但2者不同之处是合并会产生一个新的提交，而变基不会，从提交图解上来看，它的提交记录更像是一个串行的，而合并更像是2条河流有一个merge节点。

我很希望文档中的一段描述：**变基就好些重新播放一样，将提交到某一分支(topicbranch)上的所有修改都移到另一分支上(basebranch)**。重放这个词很准确地说出了rebase变基所做的事情。

文档分析

  • git rebase basebranch

变基时，会找到2个分支（topicbranch experiment和基底分支basebranch master)的共同祖先C2，然后对比当前分支(experiment)的历次提交，提取相应的修改并存为临时文件，然后将当前分支指向目标基底C3，最后以此将之前另存为临时文件的修改依序应用。以上是官方解释，有点饶，简单点说，就是把自动提交差异放到共同祖先提交节点后，当然如果和基底分支提交有冲突confilt。是需要手工处理的。

然后对master分支快进合并**

再放一张直接Merge的图式：

看看C5节点，就是自动生成的新的快照。

  • git rebase onto

取出另外分支的提交，简单说，就是将非直接checkout的分支的提交合并。

如从master上分出server分支，再从server分支分出client分支。

如果master只需要C8,C9，而不想合并C3，那么如何操作？

以上命令的意思是：“取出 client 分支，找出它从 server 分支分歧之后的补丁， 然后把这些补丁在 master 分支上重放一遍，让 client 看起来像直接基于 master 修改一样”。这理解起来有一点复杂，不过效果非常酷。

这需要对自己的每次提交的内容得非常清楚！

  • 注意！！！

如果提交存在于你的仓库之外，而别人可能基于这些提交进行开发，那么不要执行变基

通俗点讲：尽量避免对已到远端的提交(origin)进行变基，一般对自己本地分支(local)使用。

因为，远端仓库，别人也可能基于这些提交进行开发，如果你执行的变基，同时强推倒远端仓库，别人pull之后，他的提交线就会混乱，同时他在push，你再pull，就会导致你的提交线会混乱。是不是很绕，有点难理解。下面结合图示将描述简单化；

场景流程如下：

1.当前状态：我通过pull远端，将teamone的提交合并，当前我在C7上。git.team1.com是远端库

2.如果teamone不想merge，不想记录C6的merge，他采用的rebase变基，然后强推push –force，这时远端及本地的状态及提交记录如下图；

远端的C4,C6消失，多了一个C4`.

本地不变。

3.这时你执行pull，将会合并2个提交历史的内容，生成一个新的合并提交。最终仓库会如图：

因为C4`是新的，而本地C4,C6并未消失(本地未执行rebase，只有teamone执行)，所以会有一个C8.

这样会导致提交异常混乱，你会发现C4和C4`的相同的提交在你的仓库里重复出现了2次。如果你再push，又会将C4,C6重新推送到服务器上去。这回就会让teamone困惑了。因为他之前rebase的操作就是要将C4和C6丢弃。

那么如何解决？

用变基解决变基(推荐)

如果团队中的某人强制推送并覆盖了一些你所基于的提交，你需要做的就是检查你做了哪些修改，以及他们覆盖了哪些修改。

Git 除了对整个提交计算 SHA-1 校验和以外，也对本次提交所引入的修改计算了校验和——即 “patch-id”。

使用git rebase，git会自动检查自己和对方的差异，再进行合并。

推荐使用 git pull –rebase而不是git pull

git pull –rebase 等于git fetch | git rebase remote/mater

同时，可以设置git config –global pull.rebase true.这样你再使用git pull时，是默认使用选项–rebase的。

  • 变基 vs 合并

​ 无所谓那种更好，关键看针对的场景和开发者的使用理念。有人认为提交历史就是记录发生什么，有人认为是项目中发发生的事，无需记录不必要的记录。

总的原则是，只对尚未推送或分享给别人的本地修改执行变基操作清理历史， 从不对已推送至别处的提交执行变基操作，像我的习惯是，针对功能性分支基本采用rebase，对于主线记录版本的分支比如main使用merge。

rebase commitID

rebase 除了可以对branch分支进行操作，还可以指定commitID来进行各种操作场景：

1.合并提交

2.修改提交

选项：

p :对commit执行选择，选择提交

r：选择提交，但是修改提交信息

e :针对提交做修订

s :选择提交，并合并到上一个提交中

f:选择提交，但只保留上一个提交的日志

其实rebase commitID从字面可以理解，也是重新变基，只是这是基于commitId为基础，所以rebase后会新建临时分支出来。

在每一个选择提交操作上，可以执行

git rebase –continue:继续变基，调到下一个commit

git rebase –abort:放弃变基

git rebase –edit-todo：继续变基，进行变基代办事项

git commit –amend : 修改提交信息

图解

先上一张功能流程图熟悉一下：

上图是一个从H5端到Native端的完整功能流程图，形成了一个完整的闭环链。其中底层（LightWeviewBridge)是已经封装好的通信引擎，业务侧无需关注，只需要关注上面业务侧的功能实现。

左边是H5端，H5端通过调用API callNative方法，发出请求Request调用Native功能，同时会注册一个回调函数，之后会回调Native端返回的数据。

右边是Native端，接受到H5端的请求后，会从配置文件bridgeConfig中获取匹配的功能，

执行原生能力，之后会将数据返回给H5端的回调函数。至此，整个功能闭环执行完成。

原理

实现JS调用Native很简单，但是如果要进一步实现JS与Native闭环互通、异步回调、JS与Native的调用标准化、统一入口等需求，就得将技术点逐个攻克。

  • JS调用Native

    在Android侧有3种方案:

    一,通过@JavascriptInterface新增原生对象映射到JS；

    二,通过WebViewClient的shouldOverrideUrlLoading()方法回调拦截url，通过自定义协议头在原生中实现。

    三,通过WebChromeClient的onJsAlert()、onJsConfirm()、onJsPrompt()方法回调拦截JS对话框alert()、confirm()、prompt()方法，对消息message进行拦截。

    LightWeviewBridge采用的第一种方案。

  • Native调用JS

    通过Webview执行js，有2种

    一，webview.loadUrl()

    二，webview.evaluateJavascript()

    LightWeviewBridge采用的混合方案。

  • 数据如何实现闭环传输

    通过约定请求体Request和返回体Response对象的数据格式，在请求和回调时分别将数据作为成员变量放入其中，在节点流程中进行绑定，帮助数据闭环流转到正确节点。

  • 异步回调

    因为JS和Java的运行环境不同，浏览器UI端JS是单线程顺序执行，而Java后台功能可多线程异步执行，所以请求与回调需要考虑异步设计。

  • 统一标准化引擎

​ 如果只考虑实现，在开发中，每一个业务都需要将以上技术重复实现一遍，所以

​ 通信引擎的设计很有必要，将端到端到出入口及数据格式标准化、统一化，帮助开发 者只关注业务功能。

代码解析

核心

  • WebViewBridge：JS调用Native的入口对象。

    callFunc(…)：暴露给JS的调用入口方法。

    call(…)：调用原生功能的统一入口方法。

  • NativeFunc：业务原生功能基础类，负责持有通信框架上下文对象，JS回调相关等。

  • IWebViewDataHandler：负责JS回调的接口，一般交给Webview实现。

配置

  • BridgeConfig：JS端到Native功能的映射配置。

模型

  • Request：JS到Native的请求体，3元素：funcID(功能唯一ID值，必填)、data(传入数据，可选)、callbackKey(回调函数ID，可选)
  • Response：Native到JS的返回体，callbackKey(回调函数ID)，data:(返回数据)

上下文

  • BridgeContext：持有Android上下文Activity和ActivityStarter，作为Native与JS界面交互功能的中间人。
  • ActivityStarter：作为进阶调用另外Activity或Fragment的接口，提供给原生功能使用。

Bridge.js

  • callToNative(…)：JS API，调用原生能力，传入参数为Request3元素。
  • callbackFromNative(…)：回调函数，供Native端调用。
  • callbacks：回调Map，保存对应回调函数key和回调函数，实现JS端异步调用，以空间换时间。

后记

LightWebviewBridge就是一个轻量级的JSBridge框架，它的出现是为了帮助那些不用集成大框架，也不想从0开始写通信实现，而且又希望将这一模块单独解耦拿出来的开发者们。

解决方案

目前市面上平台APP中的H5应用(如小程序、快应用、流应用…)，这类程序或应用的概念的实现大都是围绕WebView而开展，实现H5与Native的通信能力，可能各自的具体细节和标准不同。

如果只需要实现H5调用原生能力，比如小程序的秒开、扫码获取信息、分享链接获取应用等众多场景，目前主流的方案要么过大，得深度集成；要么只是一个实现介绍，具体得自己从头实现，而与业务侧深度绑定；

有没有一个即能快速集成，又将H5-Native通信的能力封装完成，而让开发者只轻松地关注业务实现，同时将公用能力模块化，LightWebviewBridge正是出于这个目的而诞生。

使用详解-基础场景

文字介绍得比较详细，尽量让闭环流程中的每个细节都解释到位，实际开发中代码比较少。

初始化

将框架上下文对象BridgeContext和webview(案列中是BaseWebView)绑定。

场景-创建二维码

这是一个典型且常用的业务场景，实现如何从H5输入数据，通过原生实现，再返回H5执行的过程。

1.Native端：

  • 类：Qr；继承NativeFunc。

  • 方法：createQr；与类一起配置到bridgeConfig.json。

  • 方法参数：data : String；从H5端的API中传入，数据格式根据业务功能约定。

  • 回调方法：callbackToJS；固定API，实现从Native端将数据返回给H5。

  • 回调方法参数：base64：String；从Native端返回，数据格式根据业务功能约定。

2.配置

文件bridgeConfig.json，位于assets目录下

  • “createQr”:调用原生能力的唯一ID
  • “methodName”：方法名
  • “className”：类名(包路径)

3.H5调用

  • callToNative：API，调用原生能力，与Native通信；三个参数，即请求体(Request)3元素。
  • “createQr”(必填)：3元素之一，原生能力的唯一ID，与bridgeConfig.json中配置对应。
  • data(可选)：3元素之二，传给Native端的数据，与Native中的*方法参数data对应。
  • function(可选) ：3元素之三，JS回调函数，接收Native端的回调数据再执行。
  • response：Native端的返回的业务数据，对应Native中的回调方法参数base64。

使用-进阶

方案2 自动配置

针对较多的仓库或用户使用，通过在全局.gitconfig配置文件中使用includeIf条件来自动加载不同的用户配置。

1、首先，创建对应仓库的配置文件

假设你有两个不同的 Git 用户，一个用于工作项目(work)，另一个用于个人项目(personal)，那就创建2个对应文件，比如.gitconfig_personal和.gitconfig_work

2、最后，配置第一步中的文件

在全局配置文件~.gitconfig中，使用 includeIf 来根据不同的条件(比如:仓库路径)加载上面的配置文件,这样就会根据不同项目或工作场景，针对gitdir指定的路径去加载对应path的配置文件，从而使用不同的 Git 用户配置，而不是默认的全局配置Git用户。

注意:

1.git仓库路径末尾需要带上/，不然可能无法识别到

2.如果一个目录下的所有仓库都对应相同的user，可以使用目录路径来替换

  • Git用户名 vs git仓库账户

Git用户名(本地)：在终端上使用Git，需要配置用户信息(用户名和email)，表示是”谁”在操作Git，来提交代码等等一系列操作。

Git仓库账户(远端):如果需要使用代码托管的git服务，比如Github，Gitee等，就需要账号信息(账号名和email)来进行注册或登录。一般使用https操作仓库时，会将账号信息配置到仓库的config中；但使用ssh操作时，并不会。

  • includeIf

    gitconfig 中的 includeIf 是 Git 2.13 版本引入的一个功能，它允许你在一个用户主配置文件中为特定的条件包含其他配置文件。这对于多用户管理以及在不同的项目或环境中使用不同的 Git 配置非常有用。通过 includeIf，你可以基于一些条件来加载特定的配置。

Android传感器介绍及使用.md

发表于 2022-09-20

传感器

传感器简介

设备上的传感器能为我们的应用添加丰富的位置和运动功能，包括 GPS 或网络位置以及加速度计、陀螺仪、温度、气压计等。设备传感器能用来测量运动、屏幕方向和各种环境条件，可以提供高度精确的原始数据。

类别vs应用场景

  • 移动(动感)：测量3个轴向上的加速力和旋转力。包括加速度计、重力传感器、陀螺仪和旋转矢量

    • 游戏中可以跟踪设备重力传感器的读数，用来推断复杂的用户手势和动作，如倾斜、摇晃、旋转或挥动

  • 位置：测量设备的物理位置。包括屏幕方向传感器和磁力计

    • 天气应用可以使用设备的温度和湿度传感器来计算和报告露点

  • 环境：测量各种环境参数，像环境气温、气压、照度和湿度。包括气压计、光度计和温度计

    • 旅行应用可以使用地磁场传感器和加速度计来报告罗盘方位

框架通用API

Android传感器框架提供了多个类和接口，帮助开发者执行各种与传感器相关的任务，利用框架可以访问设备上提供的传感器并获取原始传感器数据。使用框架可以执行以下操作：

  • 获取设备上的传感器列表
  • 获取指定传感器的属性：例如最大量程、制造商、功率和分辨率
  • 获取原始传感器数据并定义获取数据的最低频率
  • 注册或取消注册用于监控传感器变化的事件监听器

框架核心

  • SensorMamanger
    提供管理传感器服务。该类提供了各种方法来访问和列出传感器，注册和注销传感器事件监听器，以及获取屏幕方向信息。还提供了几个传感器常量值，用来报告传感器的精度，设置数据采集频率和校准传感器。开发者可以使用这个类来创建传感器服务的实例。

  • Sensor

      传感器。使用这个类来创建特定传感器的实例。该类提供了各种方法来确定传感器的特性。

  • SensorEvent

      传感器事件。系统使用这个类来创建传感器事件对象，该对象提供有关传感器事件的相关信息，例如：原始传感器数据、生成事件的传感器类型、数据的准确度和事件的时间戳。

  • SensorEventListener
    接口，可创建两种回调方法，以在传感器值或传感器精确度发生变化时接受通知(传感器事件)。

常规操作

流程

获取服务->获取传感器->注册监听(处理事件)->注销

  • 识别传感器

  • 传感器特性

  • 监听传感器事件

注意事项

  • 使用前台访问

  • 取消注册传感器监听器

  • 不要阻塞onSensorChanged()方法

传感器数据可以以高频率变化，这意味着系统可能频繁调用onSensorChanged(SensorEvnent)方法。最好是在这个方法中执行尽可能少的任务，以免阻塞它。如果需要过滤后删减传感器数据，应该在此方法外执行该任务。

  • 避免使用已弃用的方法或传感器类型

  • 使用传感器之前应先进行验证

  • 谨慎选择传感器延迟

其他

传感器事件数据(SensorEvent)

  • 运动传感器：
    • 数据：运动传感器为每个SensorEvent返回传感器值的多维数组。像在单个传感器事件期间，加速度计返回三个坐标轴的加速力数据，而陀螺仪返回三个坐标轴的旋转速率数据。这些数据值与其他SensorEvent参数一起在float数组中(values)返回。
    • 场景：旋转矢量和重力传感器是运动检测和监控的最常用传感器。选择矢量可用于各种与运动有关的任务，例如检测手势、监控角度变化、监控相对屏幕方向变化。AOSP提供了以下3个基于软件的运动传感器：重力监控器、线性加速和旋转矢量传感器。所有这3个传感器都依赖于陀螺仪。
    • 重力：提供重力方向和大小的三维矢量。通常用于确定设备在空间中的相对屏幕方向。单位(m/s2)。坐标系与加速传感器使用的坐标系相同。
    • 线性加速计：三维矢量，表示沿着每个设备轴的加速度（不包括重力）。可以用于执行手势检测，还可以用作惯性导航系统的输入数据，该系统使用航位推测法。单位(m/s2)
    • 选择矢量：旋转矢量将设备的屏幕方向表示为角度和轴的组合，其中设备已围绕设备(x,y或z)旋转了多少度。选择矢量元素没有单位。坐标使用世界坐标系。X定义为矢量积Y x Z。其在设备当前位置与地面相切，并大约指向东。Y在设备当前位置与地面相切，并指向地磁北极。Z指向天空并与地平面垂直。
  • 位置传感器：地磁场（硬件）、加速度计 以及近程传感器（硬件）
    • 简介：位置传感器在确定设备在世界参考系的物理位置很有作用。例如，您可以结合使用地磁场传感器和加速度计来确定设备相对于磁北极点的位置。您还可以使用这两种传感器，在应用的参照系中确定设备的屏幕方向。位置传感器通常不会用于监测设备的移动或运动情况，例如晃动、倾斜，或冲击（详情请参阅运动传感器）。
    • 数据：地磁场传感器和加速度计会为每个 SensorEvent 返回传感器值的多维数组。例如，地磁场传感器提供单个传感器事件中所有三个坐标轴的地磁场强度值。同样，加速度计传感器测量传感器事件中施加到设备的加速度。如需了解有关传感器所用坐标系的详细信息，请参阅传感器坐标系。近程传感器会为每个传感器事件提供一个值。表 1 总结了 Android 平台支持的位置传感器。
    • 游戏旋转矢量传感器：
    • 地磁旋转矢量传感器：
    • 设备的屏幕方向：通过计算设备的屏幕方向，您可以监测设备相对于地球参照系（具体为磁北极）的位置。**计算逻辑**：使用地磁场和加速度计来计算屏幕方向的角度，通过2个硬件传感器，系统可以提供以下3个屏幕方向角度数据：
      • 方位角（绕z轴旋转角度）：此为设备当前指南针方向与磁北向之间的角度。如果设备上边缘朝北，则为0度；朝南，则为180.朝东，90；朝西，270；范围0~360。
      • 俯仰角（绕x轴旋转角度）：此为平行于设备屏幕的平面与平行与地面的平面的角度。如果设备与地面平行放置，且下边缘最靠近用户，同时将设备边缘向地面倾斜，叫俯仰角变为正值。相反方向倾斜(设备上边缘向远离地面方向移动)，角度变为负值。值的范围：-90~90
      • 倾测角（绕y轴旋转角度）：此为垂直于设备屏幕的平面与垂直于地面的平面之间的角度。如果将设备与地面平行放置，且其下边缘最靠近用户，同时将设备左边缘像地面倾斜，则角度变为正。沿相反方向倾斜(将设备右边缘移向地面)将使角度变为负值。范围-90~90
  • 环境传感器：4种传感器，光、温度、湿度和气压

针对不同场景下的获取不同的传感器的数据的不同，开发者应该针对通过框架获取的数据进行相应的处理；比如：天线测量中的方位计算。

坐标系

  • 设备参考系
    传感器框架使用标准的3轴坐标系来表示数值。设备处于默认屏幕方向时，会相对于设备屏幕来定义坐标系。X轴水平向右，Y轴垂直向上，Z轴垂直与屏幕向外。这些传感器使用此坐标系，像加速度、重力、陀螺仪、线性加速度、地磁场。

    特别注意的是：当设备的屏幕方向改变时，坐标轴不会转换，也就是说，传感器的坐标系不会随着设备的移动而改变。此行为与 OpenGL 坐标系的行为是相同的。还需要注意的一点是，您的应用不能假设设备的自然（默认）屏幕方向是竖屏。许多平板设备的自然屏幕方向为横屏。传感器坐标系始终基于设备的自然屏幕方向。

    最后，如果您的应用将传感器数据对应到屏幕显示，您需要使用 getRotation() 方法确定屏幕的旋转度，然后使用 remapCoordinateSystem() 方法将传感器坐标映射到屏幕坐标。即使您的清单指定了仅限竖屏显示，您也需要这样做。

  • 世界参考系
    有些传感器和方法使用的坐标系是基于世界参考系。这些返回的数据表示设备相对于地球的运动或位置。如需了解详情，请参阅 getOrientation() 方法、getRotationMatrix()方法、屏幕方向传感器和旋转矢量传感器。
